Introduction

GUI application based on wxWidgets to explore formulas, structures and proofs.

  • Syntax and semantics based on Nortmann's DL³ system. A modal extension of classical propositional logic and the alethic logic S5.
  • Symbolic representation of DL-formulas similar to their visual representation in LaTeX, internally using Scalable Vector Graphic (SVG) rendering.
  • Abstract syntax tree generation of DL-formulas using an Earley parser with a custom Context-free grammar.
    • Calculate pretty layouts for abstract syntax trees via the Reingold–Tilford algorithm, and compose them into SVGs.
    • View generated 'geometry only' SVGs, which are rendered using code of the library igagis/svgren and its dependencies (all MIT-licensed).
  • DL-structures:
    • Decide definition properties
    • DL-formula evaluation and obligation verification
    • View, edit, count
    • Generate lists of representatives and random samples
  • DL-proofs:
    • Integrated editor for proofs based on conditional syntactic consequence





    • Validity checks, including conditional validation with custom axioms
    • Proof unfolding and reduction
    • LaTeX code generation
    • A Metamath executable output interpreter to load proofs
    • Metamath parser and converter for .mm files

Building

Eclipse CDT project files which contain all the build settings that are exemplary used in the following have been released at releases/results_v2/EclipseProjectData.7z.

Since the Boost libraries are header-only, there are only 3 dependencies which require binaries:

  1. wxWidgets
  2. GMP or MPIR
  3. Cairo

I have only built proof-tool on Windows, using standalone builds of GCC and MinGW-w64 as compilers, such as available on winlibs.com. Since the dependencies tend to be created for building on Linux, I expect that to be far easier on Linux distributions since the dependencies can be 'just built' and used. This is not the case on Windows systems. One typically relies on binaries that were cross compiled for Windows or on people who write building scripts that actually work on Windows systems.

However, in order to link statically (e.g. to build a standalone binary), one must have static libraries of the dependencies that are compatible with the used compiler, which means the static libraries have to be built by the same compiler in a similar version. This usually means, in order to link statically, one has to also build some of the dependencies.

Building on Windows

When writing this, the sole source of recent GCC compilers released for Windows known to me is brechtsanders/winlibs_mingw, most recently GCC 11.2.0 + MinGW-w64 9.0.0 MSVCRT (release 6) (32 bit, 64 bit). I don't know whether or how proof-tool can be built with MSVC, but I am free to publish tutorials that I receive from people who figure it out. (The same goes for different plattforms.)

I managed to build and test wxWidgets versions 3.1.0 up to 3.1.6 with MinGW-w64. For wxWidgets 3.1.0 and 3.1.1, there are missing features leading to undesired behaviour, and wxWidgets 3.1.4 – 3.1.6 have significant rendering performance issues for wxGrid components on MSW, making them useless for Windows builds. Therefore, until wxWidgets' contributors fix their known issue (likely never as elaborated at releases/statdeps_v3 and releases/dyndeps_1), only wxWidgets 3.1.2 and 3.1.3 are suitable.

Unix-like Environment on Windows

It has proven useful to me to make shell scripts – .sh files designed to run on Unix terminals – applicable on Windows. For that, I installed Mintty — Cygwin Terminal emulator and executed the following in a registry (.reg) file:

Windows Registry Editor Version 5.00
[HKEY_CLASSES_ROOT\Applications\mintty.exe\shell\open\command]
@="\"D:\\cygwin64\\bin\\mintty.exe\" -w max -h always -t \"%1\" -e /bin/bash -li -c 'cd \"$(dirname \"$(cygpath -u \"%1\")\")\" && bash -x \"$(cygpath -u \"%1\")\"'"
; This is a simple registry file to automate the execution of sh via cygwin on Windows 7, might work on other Windows versions ... I did not test.

Note that CR LF is not a valid line ending in .sh files. There are editors such as Notepad++ to properly view and edit them on Windows.

Static Build on Windows: Preliminaries

Static libraries sufficient to create both 32 bit and 64 bit standalone binaries are provided at releases/statdeps_v3, which includes building instructions.

Dynamic Build on Windows: Preliminaries

Compatible dynamic libraries and building instructions are provided at releases/dyndeps_1. There are no official wxWidgets 3.1.3 posix-dwarf (32 bit) MinGW-w64-compatible releases, so I built them myself. The process also worked for wxWidgets 3.1.3 posix-seh (64 bit) binaries, which are thereby available as custom builds as well.

For dynamic builds, I recommend to always have the required DLLs (including those shipped by the compiler) present in the directory of the executable. For example, having some DLLs missing while an inadequate version is still loaded by the system (e.g. when an equally named library occurs in the PATH system environment variable) can cause confusion. For such issues I recommend using shell scripts like

objdump -p ProofTool.exe | grep "DLL Name"

to list dependencies of an .exe or .dll file, or

IFS=:
for p in $PATH ; do
find "$p/" -maxdepth 1 -name 'libstdc++-6.dll'
done

to locate redundant libraries. (I explained how to use shell scripts on Windows under Unix-like Environment on Windows.)

GCC Compiler Options

Include paths vary when using dependencies of different versions, or when not on a Windows system. The options given below are correct when following the instructions from Static Build on Windows: Preliminaries or Dynamic Build on Windows: Preliminaries.

Consider ${PROJECT_HOME} the path of proof-tool's home folder, and ${LIBS_HOME} a path where dependencies are stored. I use Eclipse, which is able to directly access these variables as defined under Project -> Properties -> C/C++ Build -> Environment. Say ${INCLUDES} is defined as follows (though that is done implicitly by Eclipse when just defining the include paths under Project -> Properties -> C/C++ Build -> Settings -> GCC C++ Compiler -> Includes).

Include Paths

-I${PROJECT_HOME}/svgren -I${LIBS_HOME}/gmp-6.1.2/include -I${LIBS_HOME}/boost_1_74_0/include -I${LIBS_HOME}/cairo-1.16.0/include/cairo -I${LIBS_HOME}/wxWidgets-3.1.3/lib/wx/include/msw-unicode-static-3.1 -I${LIBS_HOME}/wxWidgets-3.1.3/include

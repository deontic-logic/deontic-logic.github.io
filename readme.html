Introduction

GUI application based on wxWidgets to explore formulas, structures and proofs.

  • Syntax and semantics based on Nortmann's DL³ system. A modal extension of classical propositional logic and the alethic logic S5.
  • Symbolic representation of DL-formulas similar to their visual representation in LaTeX, internally using Scalable Vector Graphic (SVG) rendering.
  • Abstract syntax tree generation of DL-formulas using an Earley parser with a custom Context-free grammar.
    • Calculate pretty layouts for abstract syntax trees via the Reingold–Tilford algorithm, and compose them into SVGs.
    • View generated 'geometry only' SVGs, which are rendered using code of the library igagis/svgren and its dependencies (all MIT-licensed).
  • DL-structures:
    • Decide definition properties
    • DL-formula evaluation and obligation verification
    • View, edit, count
    • Generate lists of representatives and random samples
  • DL-proofs:
    • Integrated editor for proofs based on conditional syntactic consequence



    • Validity checks
    • Proof unfolding and reduction
    • LaTeX code generation
    • A Metamath executable output interpreter to load proofs
    • Metamath parser and converter for .mm files

Building

Eclipse CDT project files which contain all the build settings that are exemplary used in the following have been released at releases/results_v1/EclipseProjectData.7z.

Since the Boost libraries are header-only, there are only 3 dependencies which require binaries:

  1. wxWidgets
  2. GMP or MPIR
  3. Cairo

I have only built proof-tool on Windows, using standalone builds of GCC and MinGW-w64 as compilers, such as available on winlibs.com. Since the dependencies tend to be created for building on Linux, I expect that to be far easier on Linux distributions since the dependencies can be 'just built' and used. This is not the case on Windows systems. One typically relies on binaries that were cross compiled for Windows or on people who write building scripts that actually work on Windows systems.

However, in order to link statically (e.g. to build a standalone binary), one must have static libraries of the dependencies that are compatible with the used compiler, which means the static libraries have to be built by the same compiler in a similar version. This usually means, in order to link statically, one has to also build some of the dependencies. Due to oddities with the officially released wxWidgets dynamic-link libraries, linking dynamically also comes with such drawbacks since unmatching compiler versions compile executables that crash on runtime with these DLLs. (Whether the crash happens on startup or only on exit depends on the exact DLL and compiler versions used.)

Building on Windows

When writing this, the sole source of recent GCC compilers released for Windows known to me is brechtsanders/winlibs_mingw, most recently GCC 10.2.0 + MinGW-w64 8.0.0 (release 1) (32 bit, 64 bit). I don't know whether or how proof-tool can be built with MSVC, but I am free to publish tutorials that I receive from people who figure it out. (The same goes for different plattforms.)

Unix-like Environment on Windows

It has proven useful to me to make shell scripts – .sh files designed to run on Unix terminals – applicable on Windows. For that, I installed Mintty — Cygwin Terminal emulator and executed the following in a registry (.reg) file:

Windows Registry Editor Version 5.00
[HKEY_CLASSES_ROOT\Applications\mintty.exe\shell\open\command]
@="\"D:\\cygwin64\\bin\\mintty.exe\" -w max -h always -t \"%1\" -e /bin/bash -li -c 'cd \"$(dirname \"$(cygpath -u \"%1\")\")\" && bash -x \"$(cygpath -u \"%1\")\"'"
; This is a simple registry file to automate the execution of sh via cygwin on Windows 7, might work on other Windows versions ... I did not test.

Note that CR LF is not a valid line ending in .sh files. There are editors such as Notepad++ to properly view and edit them on Windows.

Dynamic Build on Windows: Preliminaries

Compatible dynamic libraries can be found:

  1. wxWidgets 3.1.4
    • 32 bit: wxMSW-3.1.4_gcc920TDM_Dev.7z
    • 64 bit: wxMSW-3.1.4_gcc810_x64_Dev.7z
  2. MPIR 2.7.2
    • Mpfr315Mpir272.zip
  3. Cairo 1.15.12
    • cairo-windows-1.15.12.zip

Only the following GCC based libraries are released as part of wxWidgets 3.1.4 (I did not consider VC variants):

GCC Version Targeted CPU Bus Width Thread Interface Exception Handling System Identifier
7.3.0 32 bit Win32 SJLJ gcc730
7.3.0 64 bit Win32 SEH gcc730_x64
8.1.0 32 bit Win32 SJLJ gcc810
8.1.0 64 bit Win32 SEH gcc810_x64
9.2.0 32 bit POSIX SJLJ gcc920TDM
9.2.0 64 bit POSIX SEH gcc920TDM_x64

There are several issues with these.

  • Mixing different exception systems leads to termination instead of handling exceptions, so we shall not mix them.
  • The posix-sjlj seems generally unable to combine with a WinLibs compiler and mpir32.dll at once (but all non-WinLibs GCC releases known to me are low-version).
  • The only posix-seh seems to be incompatible with some of the other chosen DLLs (such that the app crashes on exit, even when using TDM-GCC itself as a compiler), or corrupted, so we cannot use it at all.
  • The win32-seh versions have compatibility issues with mpir64.dll when using a WinLibs version lower than GCC 10.2.0 + MinGW-w64 7.0.0 (release 2) (fortunately they fit together for higher versions).
  • There is currently only one recent SJLJ capable WinLibs release, and it is not compatible with the win32-sjlj versions. TDM-GCC provides the most recent alternative with SJLJ that I found.

Using POSIX threads while linking towards native Win32 thread libraries seems to work fine. But due to all the issues I have encountered with data-link libraries, I would not recommend using them if you prefer to run a stable program. For example, having some DLLs missing while an inadequate version is still loaded by the system (e.g. when an equally named library occurs in the PATH system environment variable) can cause confusion. For such issues I recommend using shell scripts like

objdump -p ProofTool.exe | grep "DLL Name"

to list dependencies of an .exe or .dll file, or

IFS=:
for p in $PATH ; do
find "$p/" -maxdepth 1 -name 'libstdc++-6.dll'
done

to locate redundant libraries. (I explained how to use shell scripts on Windows under Unix-like Environment on Windows.)

For 32 bit I matched gcc920TDM with compiler TDM-GCC-32 9.2.0, and for 64 bit I matched gcc810_x64 with the currently most recent WinLibs compiler GCC 10.2.0 + MinGW-w64 8.0.0 (release 1). Note that using only GCC 9.2.0 for 32 bit comes with the drawback that only -std=c++17 -fconcepts can be used as a dialect, rather than -std=c++20.

There is a custom TDM-GCC option -shared-libstdc++ to not apply the static insertion, but it does not seem to work. When using this flag, the resulting executable crashes on exceptions and on exit.
